# Product

## Register

product

## Users

Developers, maintainers, and technical teams use Awesome while evaluating GitHub projects that appear across curated awesome lists. They are usually comparing repositories, checking project health, importing their own GitHub context, or returning to watched areas of interest.

## Product Purpose

Awesome makes scattered awesome-list recommendations searchable and comparable in one place. Success means users can find maintained, relevant repositories faster, understand why a project is worth attention, and monitor GitHub activity without leaving the product.

## Brand Personality

Clear, trustworthy, and practical. The interface should feel like a focused developer tool: calm enough for repeated use, explicit about data freshness, and direct about what each control does.

## Anti-references

Avoid marketing-heavy SaaS dashboards, decorative account pages, novelty navigation, and security surfaces that imply unsupported auth options. Do not make the product feel like a generic admin template or an over-styled analytics console.

## Design Principles

- Keep repository discovery and monitoring workflows close to the user action that starts them.
- Make GitHub the obvious account anchor, since it is the supported authentication path.
- Reserve strong visual emphasis for primary repository workflows, import state, and destructive actions.
- Prefer readable status summaries over dense settings forms.
- Leave room for future watch and newsletter preferences without exposing non-working controls.

## Accessibility & Inclusion

Target WCAG AA contrast for text, controls, focus states, and alerts. Keep controls keyboard-accessible, respect reduced motion, and avoid communicating state through color alone.
